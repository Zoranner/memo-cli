# Memo 命令设计哲学

**日期：** 2026-04-21  
**范围：** CLI 公开命令面的设计原则、命名语义与行为边界

## 文档目的

本文档只回答一个问题：Memo 的公开命令体系应当如何表达，才符合这套记忆系统本身的设计哲学。

它不是迁移说明，不是兼容方案，不是实施计划，也不是旧命令对照表。本文档关注的是公开命令面应当遵守的原则，以及这些原则在命名和默认行为上的落点。

## 核心判断

Memo 不是通用数据库工具，也不是索引运维工具。它对用户暴露的，不应是底层实现细节，而应是记忆系统的动作语义。

因此，公开命令面不应围绕以下概念组织：

- 数据摄入
- 查询语句
- 索引刷新
- 任务队列
- 内部调度

公开命令面应围绕以下概念组织：

- 唤醒记忆空间
- 记住内容
- 回忆内容
- 反观单条记忆
- 整理与巩固记忆
- 了解系统当前状态
- 在必要时恢复派生层

这意味着，Memo 的命令设计首先是记忆哲学的表达，其次才是实现能力的包装。

## 设计目标

Memo 的公开命令体系应同时满足以下目标：

- 让用户面对的是“记忆动作”，而不是“工程动作”；
- 让高频动作足够少、足够稳、足够容易记住；
- 让命令之间的语义边界清晰，不出现职责重叠；
- 让默认行为尽可能承担轻量工作，减少用户维护负担；
- 让少量重型动作保留显式入口，但不主导产品心智。

## 设计原则

### 公开命令只表达用户意图

公开命令应直接表达用户想做什么，而不是系统内部如何完成。

用户真正关心的是：

- 我想把一段内容记住；
- 我想回忆某个主题；
- 我想看清某条记忆；
- 我想让系统整理一下记忆；
- 我想知道系统现在怎么样；
- 我想在必要时恢复派生状态。

用户通常并不关心：

- 写入经过了几段流水线；
- 哪些索引被标记为 pending；
- 后台任务是立即执行还是排队执行；
- 当前处于哪个 memory layer；
- 当前 support 来自哪个 session。

因此，这类内部概念不应成为公开命令面的中心词汇，也不应进入高频参数设计。

### 一个动作对应一个命令

同一类用户意图只应对应一个公开命令。

如果一个动作需要用户在多个命令之间做选择，或者需要理解多套近义词，说明命令面已经泄露了实现分层。

命令设计应避免以下问题：

- 同一类写入动作被拆成多个入口；
- 同一类整理动作被拆成多个入口；
- 恢复动作和整理动作混成同一个入口；
- 查看状态与查看内容共用同一命令心智。

一套稳定的命令体系应让用户在第一次学会之后，能够靠直觉重复使用，而不需要反复查文档确认术语差异。

### 命名应统一在同一语义宇宙中

公开命令一旦选择“记忆系统”作为产品心智，命名就应保持在同一套语义宇宙中，不应一半是工程术语，一半是哲学术语。

因此，推荐命名方向应集中在以下词汇：

- `awaken`
- `remember`
- `recall`
- `reflect`
- `dream`
- `state`
- `restore`

这一组词的作用不是追求文学性，而是让用户在看到命令时，立刻知道自己是在与一个“记忆系统”交互，而不是在直接操作数据库、索引器或任务队列。

### `dream` 是记忆生命周期动作，不是修复动作

`dream` 必须保持为一等概念。

原因不是情怀，而是因为它准确表达了系统中的一类独立动作：记忆整理、巩固、冷却、归并、冲突收口，以及生命周期推进。

这类动作有两个重要特征：

- 它们改变的是记忆的组织与权重关系；
- 它们不等同于修复系统故障。

因此，`dream` 不能被压扁为通用 maintenance、fix 或 repair 语义。那会让“整理记忆”和“修系统”重新混成一团。

应明确区分：

- `dream` 负责记忆整理；
- `restore` 负责派生层恢复。

这两类动作都重要，但它们不是同一件事。

### 轻量工作自动处理，重型动作显式暴露

命令面不应把每一个内部步骤都变成用户负担。

高频主路径中，系统应自动完成合理的轻量工作，例如：

- 真相源写入；
- 最小即时可用状态更新；
- 必要的轻量后处理；
- 后续重型工作标记。

而以下动作应保留为显式入口：

- 明确进行一次记忆整理；
- 明确进行一次更完整的 dream；
- 明确恢复派生层；
- 明确执行全量恢复。

这样做的目的，是让“记住一条内容”仍然是轻的，而不是让每次写入都裹挟整个后台系统。

## 推荐公开命令体系

```bash
memo awaken [path]
memo remember <content> [--time <rfc3339>] [--entity <type:name> ...] [--fact <subject:predicate:object> ...] [--dry-run] [--json]
memo recall <query> [-n <limit>] [--deep] [--json]
memo reflect <id> [--json]
memo dream [--full] [--json]
memo state [--json]
memo restore [--full] [--json]
```

## 命令语义

### `memo awaken`

用于唤醒并初始化一套本地记忆空间。

它代表的是“让记忆系统进入可用状态”，而不是单纯的工程初始化命令。它应负责创建运行所需的本地环境和基础配置。

### `memo remember`

用于记住新的内容。

这是唯一的公开写入入口。它应承接原始文本输入、必要时的结构化补充，以及写入前预览等需求，但不应把内部生命周期控制参数暴露给用户。

用户表达的是“记住这个”，而不是“以某层级、某会话、某内部策略摄入数据”。

### `memo recall`

用于回忆相关记忆。

它表达的是“把相关内容找回来”，而不是执行一个数据库式查询。用户面对的是回忆动作，系统内部如何组合 exact、alias、BM25、vector、graph、rerank 等信号，应被隐藏在实现层。

`--deep` 可以保留为少量显式强度开关之一，因为它仍然对应用户可理解的差异：做一次更深的回忆。

### `memo reflect`

用于查看并理解单条记忆。

它与 `recall` 的区别在于：`recall` 解决“找什么”，`reflect` 解决“认真看哪一条”。因此它应是一个聚焦于单项详情的动作，而不是工程调试入口。

### `memo dream`

用于整理、巩固并推进记忆生命周期。

`dream` 的职责在于：

- 让零散记忆逐渐形成稳定结构；
- 推进强化、晋升、冷却和冲突收口；
- 在必要时消费与 dream 相关的排队整理工作；
- 刷新与记忆整理直接相关的轻量状态。

`dream --full` 代表更完整的一次整理，而不是派生层全量重建。

### `memo state`

用于查看当前系统状态。

这里的“状态”应是用户可理解的系统摘要，例如总量、分层、健康状况、是否存在待处理维护等，而不是默认吐出一屏实现指标。

### `memo restore`

用于在必要时恢复派生层。

它存在的意义，是保留一个明确的恢复入口，让用户在派生状态不一致时可以主动修复。它不应抢占主心智，只应在必要场景出现。

`restore --full` 代表基于真相源进行全量恢复。

## 参数哲学

参数设计应遵守同样的最小暴露原则。

### 应保留的参数类型

公开命令可以保留三类参数：

- 用户主动提供的输入补充，例如 `--time`、`--entity`、`--fact`；
- 少量用户可理解的强度开关，例如 `--deep`、`--full`；
- 机器消费所需的输出控制，例如 `--json`。

### 不应保留的参数类型

以下类型的参数不应进入公开主命令面：

- 存储层策略参数，例如 `--layer`；
- 内部支持证据参数，例如 `--session`；
- 索引实现细节参数，例如 text/vector scope；
- 队列和调度细节参数，例如 trigger、enqueue、job limit。

原因很简单：这些参数表达的是系统如何工作，而不是用户想做什么。

## 默认行为哲学

### `remember` 的默认行为

`remember` 默认应完成真相源写入与最小即时可用工作，并在必要时把更重的后续工作标记出来。

它不应在默认情况下自动触发重型整理或全量恢复，因为那会破坏写入动作的轻量属性。

### `recall` 的默认行为

`recall` 默认应走正常检索路径，并允许系统在内部根据策略自动升级。显式 `--deep` 用于用户主动要求更深检索，而不是暴露更多内部旋钮。

### `dream` 的默认行为

`dream` 默认应执行常规记忆整理。`dream --full` 应执行更完整的一次整理，但仍然属于记忆生命周期动作，而不是恢复动作。

### `restore` 的默认行为

`restore` 默认应以保守方式恢复派生状态。`restore --full` 才对应全量恢复。

## `dream` 的定时执行哲学

`dream` 既应当保留为显式人工命令，也应允许被系统定时自动执行。

这不是例外，而是符合其本质的正常机制。因为 `dream` 表示的是周期性记忆整理，而不是故障修复。

因此，以下判断应被明确写入产品心智：

- 定时 `dream` 是正常且健康的；
- 定时 `restore --full` 不是正常行为。

如果系统长期依赖周期性全量恢复才能维持可用，说明真正需要被修正的是恢复边界或派生层设计，而不是把重建当成日常动作。

## 应避免的命令设计错误

一套偏离本文哲学的命令体系，通常会出现以下问题：

- 同时暴露多套近义命令，让用户难以判断入口；
- 让索引、队列、层级这些实现概念进入主路径；
- 把记忆整理与故障修复混为一谈；
- 把临时过渡能力当成长期公开概念；
- 在命名上混用工程词和哲学词，导致整体风格失衡。

这些问题并不只是命名不好看，而是会直接破坏用户形成稳定心智模型的能力。

## 结论

Memo 的公开命令体系应当让用户面对一套稳定、统一、可解释的记忆动作语言，而不是一组暴露内部实现的工程命令。

这套语言的核心不是“系统有哪些能力”，而是“用户与记忆系统之间发生了哪些动作”。

因此，命令设计的根本标准应是：

- 是否表达了记忆动作；
- 是否减少了用户选择成本；
- 是否守住了整理与恢复的语义边界；
- 是否让默认行为承担了合理的轻量工作；
- 是否让整个命令面保持在同一语义宇宙中。

只有满足这些条件，命令体系才真正服务于产品心智，而不是继续复制当前实现结构。
