# Memo-Brain Roadmap

> 基于 `NEXT.md` 与当前代码实现的对照整理。这个文档不是产品宣传，而是执行路线图。状态判断以当前仓库实现为准。

## 目的

`NEXT.md` 定义了 memo-brain 下一阶段的目标形态：纯本地、单程序、低延迟、分层记忆引擎。

当前仓库已经完成了不少基础能力，但还没有到最终形态。这个文档的目标是把现状、缺口和后续阶段收敛成一条可执行路线，避免继续在旧产品形态和新引擎形态之间摇摆。

## 当前快照

当前实现已经具备这些核心能力：

- `SQLite` 真相源已经落地，`episodes / entities / facts / edges / memory_layers / index_state` 这些核心结构均已建模。
- `Tantivy` 文本索引已经接入，支持重建和查询。
- 查询链路已经具备 `L0 / L3 / exact / alias / BM25 / vector / graph hop / recency / layer boost / hit boost / MMR` 这些信号。
- deep search 已经具备基础自动升级能力，并可在配置了 rerank 时触发精排。
- `L1 / L2 / L3` 分层与 dream 基础规则已经存在，包含晋升、归档、冲突失效和 L3 冷却。
- 嵌入、实体抽取与 rerank 已经可以通过 provider 配置接入远端能力。
- 向量检索已经具备持久化 `HNSW` 索引能力，仍保持“索引是派生层、SQLite 是真相源”的结构。

当前实现仍有这些明显限制：

- 当前向量能力虽然已接入持久化 `HNSW` 索引，但距离 `NEXT.md` 里的最终本地 ANN 路线还没有完全收口。
- 默认能力仍依赖远端 provider，不是 `NEXT.md` 目标中的本地 `ONNX Runtime`。
- rerank 虽已接通配置与检索流程，但仍不是本地优先、按预算严格调度的最终形态。
- 写入和索引更新仍是同步热路径，不是异步任务式流水线。
- dream 的公开边界已经收口到显式命令与外部系统调度；后续重点是继续简化运行时实现，而不是回头引入内建后台系统。
- 模块边界仍偏粗，`engine.rs` 承担了过多 orchestration 责任。
- 对外文档收口仍未完成，README 仍需要继续和当前 engine 化实现对齐。

| L0/L1/L2/L3 分层 | 部分完成 | L0 较轻，L1-L3 已具备基础机制 |
| Dream | 部分完成 | 已有整理规则与显式命令入口，运行责任已收口到显式调用或外部调度 |

## 路线原则

后续推进遵守这些原则：

- 优先完成影响架构方向的缺口，不优先做表层 CLI 包装。
- 优先把“假能力”替换成“真能力”，例如线性扫描替换成 ANN，而不是继续围绕现状加补丁。
- 不把远端 provider 路径当成最终形态，只把它当迁移期能力。
- 每个阶段结束时都要能独立验证，不做“先堆一堆半成品再统一收口”。

## 阶段路线

### Phase A

目标：把当前骨架固化成稳定的 engine baseline，为后续替换底层能力做准备。

范围：

- 继续补齐 `memo-engine` 测试，把当前检索与 dream 行为锁死。
- 梳理查询路径中的评分信号和阈值，避免后续替换底层索引时行为漂移不可控。
- 收敛 README、COMMANDS 与当前 CLI/engine 实现的差距，明确当前不是旧版 `embed/search/update` 产品模型。

完成标志：

- 关键检索与晋升规则都有回归测试。
- 对外文档不再误导读者进入旧架构心智。
- 当前引擎行为可以被清晰描述，而不是只能靠读源码理解。

### Phase B

目标：把向量检索从“能跑”升级到“架构正确”。

范围：

- 用真实 ANN 结构替换当前 JSON + 全量余弦扫描实现。
- 优先选择 `USearch`，如集成成本或平台约束不合适，再评估 `hnsw_rs`。
- 保持“向量索引是派生层，不是真相源”，命中后仍回 SQLite 取记录。
- 保留索引全量重建能力，确保索引损坏时可以从 SQLite 恢复。

完成标志：

- `vector_index` 不再把所有向量全读入 `HashMap` 后线性扫描。
- 向量查询与增量 upsert 的接口稳定。
- 现有 recall 流程无需了解底层 ANN 细节。

### Phase C

目标：把模型执行从远端 provider 迁到本地优先。

范围：

- 引入 `ONNX Runtime` 路径，用于本地 embedding。
- 为 rerank 预留本地实现接口，但只在需要时触发，不把它放到默认热路径。
- 评估 extraction 的本地化路径；若短期无法完全替换，至少要把远端依赖明确标记为过渡方案。
- 在配置层区分“本地优先”和“兼容远端”两类 provider。

完成标志：

- 在没有远端 API 的情况下，最小检索闭环可以本地运行。
- embedding 不再强绑定 `lmkit` 远端 provider。
- provider 配置语义清晰，不再默认把远端当成标准路径。

### Phase D

目标：把写入和索引更新从同步热路径中拆出来，并收口 dream 的运行时边界。

范围：

- 将写入路径拆成“写 SQLite + 更新 L0”和“索引更新任务”两段。
- 保持 `memo dream` 为显式入口，不再把内建后台调度当作目标能力。
- 保持运行时实现围绕显式调用展开，避免再次引入 `dream_jobs` 这类内部排队心智。
- 为索引更新失败重试、重建和损坏恢复建立最小机制。

完成标志：

- `remember` 不再同步承担所有索引更新工作。
- 索引更新失败不会破坏真相源。
- dream 的运行责任清晰停留在显式命令或外部系统调度，不再内建一套后台任务心智。

### Phase E

目标：收敛模块边界，让工程结构和 `NEXT.md` 的目标一致。

范围：

- 从当前 `engine.rs` 中拆出更清晰的子模块，例如 `retrieve / dream / session / nlp / storage / index_text / index_vector`。
- 缩小单文件责任，避免查询、写入、dream、session cache 继续堆在同一个 orchestrator 中。
- 保持外部 API 稳定，内部逐步重构。

完成标志：

- `engine.rs` 不再承担大部分核心逻辑细节。
- 各模块职责清晰，能单独测试。
- 后续继续增强 graph、rerank、本地模型时不需要继续放大一个中心文件。

### Phase F

目标：完成 `NEXT.md` 中真正的“快路径优先、深搜按需升级”。

范围：

- 将当前显式 `deep` 开关升级为策略驱动。
- 根据置信度、候选分数接近程度、主题跨度决定是否扩大候选池、增加 hop、触发 rerank。
- 明确热路径预算和降级策略，避免默认回到重链路。

完成标志：

- 默认查询只走轻路径。
- 深搜触发有明确规则，不依赖人工参数。
- 结果质量提升来自策略，而不是单纯放大候选池。

## 近期优先级

如果只看接下来最值得做的几件事，优先顺序建议是：

- 把向量索引替换成真实 ANN。
- 接通 rerank 的配置与查询链路，但保持按需触发。
- 推进本地 `ONNX Runtime` 路线，降低远端 provider 依赖。
- 再推进本地 `ONNX Runtime` 和模块边界重构。

原因：

- ANN 和异步索引模型决定这条路线是不是“真引擎”，优先级高于继续优化旧式 provider 路径。
- 本地模型接入很重要，但如果索引更新和运行时边界还是旧形态，会把复杂度堆到错误位置。

## 暂不做

这些内容不应在当前阶段抢优先级：

- 分布式、多进程、高可用部署。
- 团队共享知识库的复杂权限和同步机制。
- 独立图数据库化。
- 再扩一套旧产品 CLI 能力来维持旧文档表象。

## 文档维护约定

这个文档用于跟踪 `NEXT.md` 的落地进度，不记录细碎任务。

建议后续配套方式：

- 阶段完成情况，更新 `ROADMAP.md`。
