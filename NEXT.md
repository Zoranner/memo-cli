# NEXT

## 定位

`NEXT` 描述 memo-brain 下一个阶段的最终方向：从当前偏 CLI 的多查询记忆工具，收敛为一个**纯本地、单程序、无外部服务依赖**的记忆引擎。

这个阶段的核心目标不是继续堆命令，而是解决三个根问题：

- 实时语音场景下，检索链路太重，响应时间不可接受
- 记忆是平面堆积，缺少分层、沉淀和长期整理机制
- 如果未来商业化，依赖带许可证风险或长期维护风险的底层组件会反噬整个产品

## 目标

- 纯本地运行，不依赖 PostgreSQL、Qdrant、Redis 等后台服务
- 单程序内完成存储、检索、索引、整理和压缩
- 以低延迟为优先，优先服务实时语音对话
- 记忆支持分层晋升，而不是所有内容直接落到同一个层级
- 真相源唯一，索引可重建
- 使用严格开源、可商业化、长期可维护的底层组件

## 非目标

- 不在这个阶段追求分布式、高可用、云原生部署
- 不在这个阶段追求大规模团队共享知识库
- 不把这个阶段做成通用数据库或独立图数据库
- 不默认依赖远程 LLM、远程 embedding 或远程 rerank

## 最终技术栈

### 主语言

- `Rust`

原因：

- 当前项目已是 Rust，延续现有工程能力成本最低
- 单程序、本地、低延迟、后台整理任务都适合 Rust
- 后续如果需要做桌面端、CLI、TUI 或嵌入宿主进程，Rust 都顺手

### 真相源

- `SQLite`
- 推荐通过 `rusqlite` 访问

职责：

- 唯一数据源
- 保存实体、关系、记忆片段、层级状态、索引状态、整理任务
- 所有向量和文本索引都只作为派生层，不作为真相源

### 文本检索

- `Tantivy`

职责：

- BM25、关键词检索、短语命中、标题/别名 boosting
- 承担人名、地点、专有名词、偏好描述的第一层高精度召回

选择原因：

- 嵌入式、Rust 原生、速度快
- 比 SQLite FTS5 更适合做长期可调优的文本检索层
- 可自定义 tokenizer，便于处理中英混合、人名、路径名、代码标识符

### 向量检索

- 进程内 ANN 索引，采用 `HNSW`
- 首选 `USearch`
- 备选 `hnsw_rs`

职责：

- 承担语义召回
- 保持完全进程内、文件型索引，不引入独立服务
- 命中后回 SQLite 取真实记录

### 本地模型执行

- `ONNX Runtime`

职责：

- 本地 embedding
- 可选轻量 rerank
- 可选实体抽取和 dream 辅助模型

约束：

- 热路径只允许小模型
- rerank 不默认常驻，只在结果模糊或 deep search 时触发

### 可选热缓存

- 先用进程内缓存，不单独引入服务
- 优先使用 `moka` 或自定义 LRU 结构

职责：

- 当前 session 的热实体
- 最近命中的记忆 id
- 最近整理出的高置信短期事实

## 总体架构

整个引擎分成五层：

- `Storage Layer`
  - SQLite 真相源
- `Index Layer`
  - Tantivy 文本索引
  - HNSW 向量索引
- `Reasoning Layer`
  - 别名解析、时间衰减、MMR、图扩展、融合排序
- `Dream Layer`
  - “做梦”机制，负责分层晋升、去重、归档、关系提炼
- `Session Layer`
  - L0 热层、短期缓存、实时语音会话态

核心原则：

- 真相只存在 SQLite
- 文本索引和向量索引都可以全量重建
- 检索结果返回前必须回 SQLite 确认真相

## 分层记忆模型

### L0 热层

特点：

- 当前会话内的短期热记忆
- 生命周期短
- 在内存中维护，不要求完全去重
- 优先服务当前轮和最近几轮对话

典型内容：

- 最近提到的人名、地点、宠物名、项目名
- 当前用户正在追问的主题
- 最近刚命中的记忆 id

### L1 浅层

特点：

- 原始 episode 候选层
- 可重复、可脏、可冲突
- 主要用于保留近期对话证据

典型内容：

- 刚刚说过的片段
- 一次性偏好
- 暂未验证的人物关系

### L2 中层

特点：

- 经 dream 整理后形成的稳定事实层
- 具备实体、别名、关系、时间归一化
- 是图扩展和结构化检索的主工作层

典型内容：

- `张伟 -> lives_in -> 杭州`
- `user -> prefers -> 手冲咖啡`
- `user -> plans_trip_to -> 上海`

### L3 深层

特点：

- 长期稳定、命中频繁、价值高
- 数量控制严格
- 启动时优先加载

典型内容：

- 核心用户画像
- 长期偏好
- 关键人物关系
- 持续性地点和经历

## “做梦”机制

“做梦”不是生成新知识，而是**记忆整理与巩固**。它负责把浅层候选整理成更深层的长期可用记忆。

### 触发时机

- 会话结束
- 程序空闲一段时间后
- 即将发生上下文压缩前
- 达到一定写入量阈值后

### 主要任务

- 清理 L0 噪声，筛出值得保留的片段写入 L1
- 对 L1 做去重、指代消解、别名合并、时间绝对化
- 从 L1 抽取实体、事实、关系并沉淀到 L2
- 将高频、高置信、跨 session 反复出现的内容晋升到 L3
- 将过时、低命中、被覆盖的内容降权或归档

### 晋升规则

建议综合以下信号：

- 被提及次数
- 跨 session 重复出现
- 被检索命中的频率
- 与高价值实体的连接强度
- 时间上的持续有效性
- 置信度

### 降级与归档

不建议硬删除。优先做：

- 降权
- 标记失效
- 归档到冷层
- 保留 provenance，保证可追溯

## 数据模型

### 核心表

- `entities`
  - 实体主表
- `entity_aliases`
  - 别名、昵称、拼写变体
- `episodes`
  - 原始对话片段或记忆片段
- `facts`
  - 结构化事实
- `edges`
  - 实体关系边
- `mentions`
  - episode 与 entity 的映射
- `memory_layers`
  - 每条记忆当前所在层级和状态
- `dream_jobs`
  - 做梦任务和状态
- `index_state`
  - 文本索引和向量索引的版本、重建状态

### 关键字段建议

- 全部业务对象使用稳定 `id`
- 保留 `created_at`、`updated_at`、`last_seen_at`
- 保留 `confidence`
- 保留 `source_episode_id`
- 保留 `version`
- 保留 `archived_at` 或 `invalidated_at`

### 图关系

不引入嵌入式图数据库。图结构直接落在 SQLite 表里：

- `edges(subject_id, predicate, object_id, weight, valid_from, valid_to, source_episode_id)`

轻量多跳用 `WITH RECURSIVE`，热点邻接表可缓存到内存。

## 检索流水线

热路径必须默认轻量，不能默认走深搜。

### 快路径

优先顺序：

- L0 热层命中
- L3 常驻记忆命中
- alias / exact 命中
- BM25 文本召回
- 向量 ANN 召回
- 1~2 hop 轻量图扩展
- 融合排序后返回

### 深搜

只在必要时触发：

- 快路径置信度过低
- top-k 分数过于接近
- 用户明确追问历史细节
- 当前主题跨层跨度明显过大

深搜可附加：

- 更大的候选池
- rerank
- 更深的图扩展
- dream 线索回查

### 排序信号

默认融合这些信号：

- exact/alias 命中
- BM25 得分
- vector 相似度
- 时间衰减
- L3/L2 层级加权
- 命中频率
- 图邻近度
- MMR 去重

## 写入流水线

### 实时写入

用户一段话进入系统后：

- 先写 SQLite episode
- 同步更新 L0
- 异步提交索引更新任务

不在热路径中做复杂 dream 整理。

### 索引更新

- Tantivy 做增量写入
- HNSW 做增量 upsert
- 如果索引异常，可根据 SQLite 全量重建

## 模块划分

建议按 crate 内模块组织：

- `core`
  - 领域模型、id、层级状态、规则
- `storage`
  - SQLite DAO、迁移、事务
- `index_text`
  - Tantivy 封装
- `index_vector`
  - HNSW 封装
- `nlp`
  - embedding、rerank、实体抽取
- `retrieve`
  - 快路径、深搜、融合排序
- `dream`
  - 做梦任务、晋升、归档
- `session`
  - L0 热层、当前会话状态
- `app`
  - CLI / TUI / SDK 对外接口

## 性能目标

本阶段不追求云上大规模吞吐，只追求单程序内的稳定低延迟。

### 目标预算

- L0 命中：`< 1ms`
- alias / exact 命中：`< 5ms`
- Tantivy BM25：`5ms - 20ms`
- 向量 ANN：`10ms - 40ms`
- 轻量图扩展：`5ms - 20ms`
- 快路径总预算：`20ms - 80ms`

说明：

- 这些是引擎内部目标，不含语音识别、LLM 生成和 UI 渲染
- 超过预算时，优先缩小候选池和跳数，而不是默认升级深搜

## 与当前实现的关系

当前 memo-brain 的核心问题是搜索链路过重：查询拆解、并行多叶子、分层扩展、远程 rerank、LLM 总结都在热路径上。

`NEXT` 的核心转向是：

- 默认不走 LLM
- 默认不走多叶子
- 默认不走远程服务
- 默认只跑快路径
- 让“做梦”承担整理和提炼工作，而不是让写入和检索阶段承担一切

## 取舍

### 为什么不用服务型数据库

- 纯本地单程序是明确约束
- 引入服务会把部署、备份、升级、分发复杂度拉高
- 对玩具型实时语音系统，单程序可交付性更重要

### 为什么不用嵌入式图数据库

- 严格开源、非 Java、长期维护、单程序、图能力完整，这几个条件交集太小
- 自己在 SQLite 上维护图表结构，反而更稳、更可控

### 为什么不用全 SQLite 扩展

- 纯 SQLite FTS/向量虽然极简，但文本检索和向量检索的调优空间不够大
- Tantivy + HNSW 作为嵌入式索引层，更适合长期演进

## 演进路径

建议按下面的顺序推进，而不是一次做满：

- 先完成 SQLite 真相源、L0/L1/L2/L3 分层和基本数据模型
- 接入 Tantivy，先把 exact + BM25 做稳
- 接入本地 embedding 和 ANN 索引
- 再补做梦机制和晋升/归档规则
- 最后再做复杂图扩展、rerank 和更强的本地模型

## 最终结论

`NEXT` 阶段的最终方案是：

- `Rust`
- `SQLite`
- `Tantivy`
- `HNSW/USearch`
- `ONNX Runtime`
- 进程内热缓存

同时采用：

- `L0/L1/L2/L3` 分层记忆
- 真相源与索引分离
- 快路径优先、深搜按需升级
- “做梦”负责整理、晋升和归档

这套方案不是最花哨的，但在“纯本地、单程序、可商业化、低延迟、长期可维护”这些目标同时成立时，是最稳的落点。
